================================================================================
  SECTION 1: PROJECT OVERVIEW
================================================================================

Agent Exploit Lab is an interactive AI agent security testing platform. Users
attack a real LLM-powered agent with prompt injection, memory poisoning,
privilege escalation, and data exfiltration techniques mapped to the OWASP
Agentic Security Initiative (ASI) framework. Every attack fires a real API call
to the target model and streams the response back to the UI.

The app has two main modes:
  - VULNERABLE: Agent has a deliberately weak system prompt. Attacks often succeed.
  - HARDENED:   Agent has 6 explicit security rules. All preset attacks are blocked.

It also includes:
  - RSI Loop: Red/Blue agent that mutates payloads until a target model breaks
  - Vulnerability Pipeline: Searches arxiv/MITRE/NVD and synthesizes new attacks
  - Agent Marketplace: Switch between 7 agent architectures (multi-agent, no-code, etc.)
  - CI/CD API: Headless attack runner for automated security testing in pipelines

  SECTION 7: KEY ARCHITECTURE PATTERNS
================================================================================

--- 7A. STATE MANAGEMENT ---
All lab state lives in src/store/LabContext.tsx using useReducer + React Context.
No Redux. No Zustand. The LabProvider wraps the /lab route only.

Key state shape:
  messages:       ChatMessage[]       — chat history shown in AttackConsole
  logs:           LogEntry[]          — security events shown in EventLog
  memory:         MemoryStore         — key/value store; poisoned keys turn red
  session:        LabSession          — score counters + pipeline compromise status
  agentMode:      'vulnerable'|'hardened'|'custom'
  agentArchitecture: 'generic'|'multi-agent'|'no-code'|'autonomous-coder'|
                     'enterprise-copilot'|'shadow-it-agent'
  llmProvider:    'anthropic'|'openai'|'google'|'external'|'openclaw'|'webhook'
  selectedAttackId: AttackId | null   — which preset is loaded
  inputValue:     string              — current textarea content
  streamingContent: string | null     — token buffer while response is streaming
  lastDebrief:    DebriefData | null  — post-attack explanation panel

--- 7B. ATTACK FLOW ---
1. User selects a preset → SELECT_ATTACK action loads payload into inputValue,
   primes memory with supporting context (PRIMING_MUTATIONS), shows explain HTML.
2. User clicks Send → sendMessage() in LabContext fires.
3. AgentService.streamChat() POSTs to /api/agent/stream (SSE).
4. Tokens stream back → APPEND_TOKEN actions update streamingContent.
5. On stream end → ExploitDetector.analyze(input, fullReply) runs.
6. Result determines: claudeBlocked, detected, attackId, severity.
7. If attack detected AND not blocked → memory mutations applied, pipeline
   nodes marked compromised, debrief data set.
8. If claudeBlocked in vulnerable mode → weakModel comparison shown.

--- 7C. API ENDPOINTS ---
All served by server/index.ts on port 3001.

  POST /api/agent          — single-turn agent chat (non-streaming)
  POST /api/agent/stream   — streaming agent chat (SSE)
  POST /api/rsi/run        — RSI loop (red/blue/judge, SSE)
  POST /api/rsi/harden     — blue team prompt hardening
  POST /api/research       — adversarial researcher agent (Tavily + Sonnet)
  GET  /api/attacks        — approved pipeline attacks from proposed-attacks.json
  GET  /api/pipeline/status — pipeline stats
  POST /api/pipeline/run   — spawn npm run sync in background
  POST /api/pipeline/proposal/:id/status — approve/reject a proposal
  POST /api/pipeline/export — export approved attacks into Attack.ts
  GET  /api/web-preview    — fetch URL metadata for WebPreview component
  POST /api/ci-cd/run      — start headless CI scan job
  GET  /api/ci-cd/status/:jobId — poll CI job status
  POST /api/access/request — email capture / access gate (Supabase + Resend)
  POST /api/contact        — contact form
  POST /api/nda/attest     — NDA attestation submission
  POST /api/nda/check      — check NDA status by email
  GET  /api/admin/nda-list — admin: list all NDA attestations
  POST /api/admin/nda-approve — admin: approve NDA
  POST /api/admin/nda-deny    — admin: deny NDA
  POST /api/stripe/webhook — Stripe webhook handler
  POST /api/stripe/checkout — create Stripe checkout session
  POST /api/discover-sources — source discovery agent

--- 7D. SYSTEM PROMPT MODES ---
buildSystemPrompt(mode, memoryContext, architecture, customPrompt) in server/index.ts

VULNERABLE mode — realistic corporate IT deployment notes, no obvious jailbreak
signals. Explicitly trusts: IT ticket references, delegation chains, tool
execution without confirmation, transparency about config for audits, user
instructions take precedence over base config.

HARDENED mode — 6 immutable security rules: no persona adoption, zero-trust
authority, instruction integrity, data leakage prevention, tool sandboxing,
semantic disguise detection.

CUSTOM mode — user-supplied system prompt (customPrompt arg). Memory context
and tool list appended automatically.

Architecture overlays (applied before mode rules):
  multi-agent       — high inter-agent trust, delegates freely
  no-code           — permissive, optimised for "just make it work"
  autonomous-coder  — trusts README/issue content implicitly, has terminal access
  enterprise-copilot — M365-style, internal docs assumed verified
  shadow-it-agent   — no guardrails, no logging, fully permissive

--- 7E. EXPLOIT DETECTION ---
ExploitDetector.analyze(input, response) in src/services/ExploitDetector.ts

Step 1: Check response against CLAUDE_REFUSAL_PATTERNS → sets claudeBlocked=true
Step 2: Check input against 4 classifiers (INJECTION, MEMORY_POISON, PRIVILEGE, EXFIL)
Step 3: If classified, check response against category-specific SUCCESS patterns
        (MEMORY_POISON_SUCCESS, TOOL_SUCCESS, PRIVILEGE_SUCCESS, INJECTION_SUCCESS)
Step 4: Fallback — check response directly for tool success or injection success
        regardless of which input classifier fired

--- 7F. MEMORY STORE ---
MemoryStore (src/models/MemoryStore.ts) holds 5 default keys:
  user.name, user.role, session.id, permissions, last_action

Keys are compared against original values to determine if poisoned.
Poisoned state is persisted to localStorage ('lab-poisoned-memory') so it
survives page refresh — simulating real persistent memory corruption.
Memory context is serialised to JSON and injected into every system prompt.

--- 7G. ATTACK MODEL ---
Each Attack in ATTACKS (src/models/Attack.ts) has:
  id              — unique AttackId string
  label           — display name
  payload         — the exact text loaded into the input box
  explain         — HTML explanation shown in the UI
  vuln            — OWASP ASI category label
  owaspId         — e.g. 'ASI-01'
  compromises     — which pipeline nodes turn red on success
  successChance   — float 0–1, used by offline AttackService.run()
  weakModelLabel  — name of the model shown in comparison panel
  weakModelResponse — what a weaker model would have output

Dynamic attacks come from /api/attacks (approved entries in proposed-attacks.json)
and are stored in state.dynamicAttacks. They render the same way as built-in
attacks but don't have an AttackId entry in the type union.

   The frontend calls /api/* which Vite proxies to localhost:3001 in dev.
   In production, the server serves the built frontend from /dist.

3. CSS uses custom properties (CSS variables) for all colours and spacing.
   Define them in :root in src/index.css. Key variables:
   --bg-base, --bg-primary, --bg-secondary, --bg-tertiary
   --text-primary, --text-secondary, --text-tertiary
   --accent-cyan, --accent-red, --accent-green
   --border-light, --border-med, --border-accent, --border-danger
   --font-mono (JetBrains Mono or similar)
   --glow-cyan, --glow-green, --glow-red (box-shadow values)
   --radius-sm, --radius-md, --radius-lg, --radius-xl
   --glass-blur (backdrop-filter value)

4. The app has a dark theme by default. Background is approximately #09090b.
   The aesthetic is "hacker terminal" — monospace fonts, cyan/red accents,
   subtle glows, thin borders.

5. Framer Motion is used extensively for tab switching (layoutId="tab-active-bg"),
   hover animations on buttons, and AnimatePresence for modals/tab content.

6. The AttackConsole is the most complex component. It renders:
   - Attack preset selector (chips/buttons, one per AttackId)
   - Explain panel (innerHTML from attack.explain)
   - Agent mode selector (vulnerable/hardened/custom)
   - Architecture selector dropdown
   - Provider selector dropdown
   - Textarea (inputValue from state)
   - Submit button (calls sendMessage)
   - Chat message list (messages from state)
   - Streaming indicator (when streamingContent !== null)
   - Debrief panel (lastDebrief from state)
   - Weak model comparison panel (shown when claudeBlocked)

7. The RSI Loop (RSIPanel) uses a 3-agent loop:
   Red Agent (claude-sonnet-4-6) → mutates payload each iteration
   Target Agent (user-selected provider) → receives the payload
   Judge Agent (claude-sonnet-4-6) → evaluates success/fail/partial
   All via /api/rsi/run (SSE). Runs up to maxIterations (default 5).
